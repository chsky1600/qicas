# QICAS (ICAS) - Instructor-Course Assignment Scheduler

QICAS is a scheduling web app for assigning instructors to course sections across an academic year (currently modeled as `Fall` and `Winter`). It consists of:

- A backend API (Express + Bun + MongoDB/Mongoose) that stores faculties, academic years, schedules, and the rule sets used for validation.
- A frontend (React + Vite) that lets users drag-and-drop assignments, automatically re-validates schedules, and shows violations ranked by severity.

The core idea: **your “schedule” is a list of assignments**, and **rules are stored per academic year**. The backend computes violations on demand; the UI surfaces them as `Info`, `Warning`, or `Error`.

## Repository Layout

At the repository root:

- `src/` - backend server code (routing, controllers, services, DB models, seed scripts)
- `tests/` - Bun test suite for core services/rules
- `client/` - React/Vite frontend application
- `dist/` - compiled backend output (generated by `bun build`)
- `package.json` - backend scripts + dependencies
- `tsconfig.json` - backend TypeScript config

### Backend: `src/`

`src/` is the backend entry for Express. The high-level structure:

- `src/index.ts` - server bootstrap (Express middleware, routes, and DB connection)
- `src/api/` - Express routers (URL-to-controller mapping)
- `src/controllers/` - request handlers (extract params/body, call services, format responses)
- `src/services/` - service layer (business logic, rule/validation computation, data operations)
- `src/db/` - Mongo connection + Mongoose schemas (mostly embedded within `Faculty`)
- `src/types/` - domain types shared across services/controllers/models
- `src/scripts/seeders/` - scripts that populate Mongo with example/smoke test data

#### `src/index.ts` (How the server starts)

This file wires together the runtime system:

1. Configures Express middleware:
   - `cors` (default permissive CORS)
   - `express.json()` for JSON request bodies
   - `cookie-parser` for reading JWT cookies
2. Mounts routers:
   - `src/api/index.ts` (health endpoint)
   - `src/api/authRouter.ts` (login/refresh/password endpoints)
   - `src/api/*Router.ts` for authenticated domains (faculty, users, courses, instructors, schedules, rules)
3. Applies authentication middleware (`verifyToken`) to the authenticated routers.
4. Connects to Mongo using `connectDB()` from `src/db/connection.ts`, then starts listening on `PORT` (default `3000`).

**Impact on system behavior:** every request (except auth + health) is expected to carry the JWT in a `token` cookie/header. That token provides `faculty_id`, which scopes all data access.

#### `src/api/` (Routes)

Routers define URL paths and HTTP methods. In practice, they act as the “public API contract” for the system.

The main routers are:

- `authRouter.ts` (authentication)
  - Handles login (sets JWT `token` cookie), session refresh, and password changes.
- `facultyRouter.ts` (faculty + year setup)
  - CRUD-ish operations for the authenticated faculty record, plus “migrate year” cloning of courses/instructors/rules (and optionally schedules).
  - Adds/removes embedded users from the faculty document.
- `userRouter.ts` (user management)
  - CRUD for users scoped to the authenticated faculty.
- `courseRouter.ts` (course data)
  - Lists and edits courses per academic year, and supports looking up where a course appears in schedules.
- `instructorRouter.ts` (instructor data)
  - Lists and edits instructor profiles per academic year, plus lookup of the instructor’s schedule assignments.
- `scheduleRouter.ts` (schedule editing + validation)
  - Manages the faculty’s “working schedule”, saved schedule snapshots, adding/removing assignments, and schedule validation (returns violations ranked by severity).
- `yearRouter.ts` (rules/config for validation)
  - Lists academic years for the authenticated faculty, and provides CRUD for course rules and instructor rules inside that academic year.

**Impact on system behavior:** routes directly determine which UI actions are possible and which payload shapes must be sent by the frontend.

#### `src/controllers/` (Request handlers)

Controllers are responsible for:

- Extracting request parameters (`req.params`) and authenticated scope (`req.body.faculty_id` set by `verifyToken`)
- Validating minimal required fields (often checking existence of IDs)
- Calling services for business logic or DB operations
- Returning JSON responses and HTTP status codes

Notable controller behavior:

- `src/controllers/authController.ts`
  - `getToken`: verifies user password using `Bun.password` and sets `res.cookie("token", jwt)`
  - `verifyToken`: validates JWT (`issuer: "qicas"`) and writes `payload.faculty_id` into `req.body.faculty_id`
- `src/controllers/scheduleController.ts`
  - CRUD is done by updating embedded subdocuments inside a single `Faculty` document
  - validation endpoint calls:
    - `validateAssignment(schedule, candidate)` when a candidate `assignment` is provided
    - otherwise `validateScheduleService(schedule)` to validate the full saved schedule
- `src/controllers/facultyController.ts`
  - `migrateFacultyToNewYear` validates request payload and delegates to `facultyService.migrateFacultyToNewYear`

#### `src/services/` (Business logic)

Service modules implement the “rules of the system” and the DB operations used by controllers.

Key services:

- `validatorService.ts`
  - `validateAssignment(schedule, candidate)` computes rule violations impacted by a candidate assignment
  - `validateSchedule(schedule)` computes all violations for the full schedule
  - Rule checks are split into:
    - `checkCourseRules(...)` (course/section-related constraints and conflicts)
    - `checkInstructorRules(...)` (instructor workload and eligibility constraints)
    - `checkScheduleRules(...)` (schedule-wide constraints such as unassigned sections and workload/section imbalance)
  - Produces a `ValidationResult` including:
    - overall `degree`: `"Valid"` or worst of `"Error" | "Warning" | "Info"`
    - `violations[]`: each with `{ type, offending_id, code, message, degree }`
- `facultyService.ts`
  - CRUD for faculty plus embedded users and embedded academic years
  - `migrateFacultyToNewYear(...)` deep-clones academic year data:
    - clones courses, instructors, rules, and optionally selected schedules
    - strips Mongo `_id` and `__v` fields to avoid shared references
    - updates cloned schedules’ `year_id` so downstream validation can resolve the correct year rules
- `userService.ts`
  - manages a separate `UserModel` collection and also synchronizes user entries into the embedded `Faculty.users` array
  - note: authentication (`authController`) looks up users inside `Faculty` by email, so the embedded data is the effective login source
- `yearService.ts`
  - `getYearConstraints(year_id)` loads the single academic year constraints from the authenticated faculty document
  - throws if the year cannot be found
- `yearRulesService.ts`
  - CRUD for course rules and instructor rules stored inside the academic year

#### `src/db/` (Mongo data shape)

`src/db/connection.ts`:

- `connectDB()` uses Mongoose to connect to MongoDB using `process.env.MONGO_URI` (default `mongodb://localhost:27017/qicas`)
- `disconnectDB()` disconnects

`src/db/models/`:

The project stores **nearly all data as a single embedded document**:

- `Faculty` is the top-level Mongo document.
- Inside it is `academic_years[]`, and each academic year embeds:
  - `courses[]`
  - `instructors[]`
  - `schedules[]`
  - `instructor_rules[]`
  - `course_rules[]`

Mongoose schemas exist for the embedded substructures (e.g., `scheduleSchema`, `assignmentSchema`), but the controllers primarily persist via `FacultyModel.updateOne(...)` and query via `FacultyModel.findOne(...)` and aggregation pipelines.

This design keeps reads for “everything related to a faculty/year” efficient while still allowing validation to load full rule context from a single document.

Each schema file corresponds to (embedded) pieces of that structure:

- `src/db/models/faculty.ts` - top-level `Faculty` schema (embeds `users[]` and `academic_years[]`)
- `src/db/models/year.ts` - `academicYearSchema` (embeds `schedules[]`, `courses[]`, `instructors[]`, `instructor_rules[]`, `course_rules[]`)
- `src/db/models/schedule.ts` - `scheduleSchema` + embedded `assignmentSchema` (used inside academic years)
- `src/db/models/course.ts` - `courseSchema` + embedded `sectionSchema` (used inside academic years)
- `src/db/models/instructor.ts` - `instructorSchema` (used inside academic years)
- `src/db/models/note.ts` - `noteSchema` (embedded inside instructors/courses depending on usage)
- `src/db/models/user.ts` - `userSchema` for the standalone `UserModel` collection (service code also syncs embedded `Faculty.users[]`)

#### `src/types/` (Domain model)

`src/types/` defines TypeScript interfaces for the domain entities used by services/controllers:

- `Faculty`
- `AcademicYear` (year-level rules + data)
- `Course`, `Section`
- `Instructor`
- `Schedule`, `Assignment`
- `InstructorRule`, `CourseRule`
- `ValidationResult`, `Violation`
- `enums.ts`: `Term`, `InstructorRank`, `CourseLevel`

**Meaning in project context:** these types define what the schedule represents and what the validation rules evaluate.

#### `src/scripts/seeders/` (How environments get data)

The seed scripts are intended for dev/test environments to create example faculties, users, courses, instructors, rules, and schedules in Mongo.

Seed scripts included:

- `seed_testing_24022026.ts`
  - connects using `process.env.MONGO_URI` (default local)
  - deletes existing `Faculty` documents
  - hashes passwords with `Bun.password.hash(...)`
  - inserts a single faculty with embedded academic year data, including a working schedule with many example “violation scenarios”
- `seed_testing_15-02-2026.ts`
  - connects using `process.env.MONGO_URI` (default local)
  - deletes existing `Faculty` documents and then creates a `FacultyModel` document with smaller sample data

**Impact on system behavior:** without seed data, the UI has nothing to list or validate; with seed data, the UI can show rule evaluation immediately.

#### `tests/` (What is verified)

Tests run with `bun test` and currently cover:

- `tests/services/facultyService.migrate.test.ts`
  - verifies deep-clone behavior when migrating academic years (no shared subdoc references, correct `start_year` increment, correct cherry-picking of schedules)
- `tests/services/validatorService.test.ts`
  - validates rule checks (course, instructor, and schedule-wide)
  - verifies the “worst degree wins” logic (`Error` > `Warning` > `Info`)

### Frontend: `client/`

`client/` is a Vite + React application that consumes the backend API.

At the client level:

- `client/package.json`:
  - `vite` dev server
  - `build` runs `tsc -b` + `vite build`
- `client/vite.config.ts`:
  - configures dev proxy routes so browser requests like `/auth`, `/courses`, `/schedule/...` reach `http://localhost:3000`
- `client/src/main.tsx`:
  - mounts the app under `react-router-dom`’s `BrowserRouter`
- `client/src/App.tsx`:
  - defines routes:
    - `/` redirect to `/main`
    - `/login` (guest-only)
    - `/main` (authenticated)

#### Authentication UX

`client/src/components/authGuard.tsx`:

- `AuthGuard`: requires presence of `token=` cookie
- `GuestGuard`: prevents logged-in users from visiting `/login`

`client/src/components/sessionWarning.tsx`:

- decodes token expiry from the JWT payload stored in `document.cookie`
- shows a modal about 5 minutes before expiry
- calls `POST /auth/refresh` to extend session
- if refresh fails, clears `token` cookie and redirects to `/login`

#### Schedule UI: `client/src/pages/schedulePage.tsx`

This is the main user-facing page:

- uses `useSchedule()` hook for all data and actions
- uses `useTutorial()` to guide onboarding with `driver.js`
- uses `DndContext` from `@dnd-kit`:
  - drag course “chips” to instructor cells (Fall/Winter)
  - drag chips back to unassign
- renders:
  - `CoursesPanel` (Courses sidepanel)
  - `ScheduleTable` (Instructor assignment main interface)
  - `PropertiesDialog` (edit instructors/courses/rules)
  - `SavedSchedulesDialog` (snapshot management)
  - `MigrationDialog` (Creating new Academic Years)
  - `UserManagementDialog` (Admin: Manage user accounts)
  - `AccountDialog` (Support: manage personal user account)
  - `HowToDialog` (How-to guides)

**How validation affects the system from the UI:**

- after assignment changes, the hook triggers a debounced `validateSchedule` call:
  - backend computes violations and returns them
  - the UI displays violations alongside schedule cells (severity-based coloring)

#### Hook-based data flow: `client/src/features/schedule/useSchedule.ts`

This hook is the “orchestrator” for schedule state:

- loads:
  - `/year` (list years)
  - `/schedule/` (working schedule metadata)
  - then per year:
    - `/courses/:year`
    - `/year/:year/rules/courses`
    - `/instructors/:year`
    - `/year/:year/rules/instructors`
    - `/schedule/:year`
- provides actions to:
  - assign/unassign sections (adds/removes assignments via schedule endpoints)
  - create/update instructors and courses
  - drop instructors/courses by updating their rules’ `dropped` field
  - create/update/delete user accounts, including setting a temp. passowrd
  - create/copy/delete/rename/switch schedule snapshots
  - change year, and migrate to new year
  - export the schedule to CSV or XLSX (default)
- computes `violations`:
  - calls `/schedule/:year/:schedule_id/validate` (schedule-wide validation)
  - stores the returned `violations[]` for rendering


#### API layer in the client

`client/src/features/schedule/api.ts` centralizes HTTP calls:

- provides strongly typed wrapper functions around backend endpoints:
  - login/password, year listing, course/instructor rule CRUD
  - schedule CRUD + assignment endpoints
  - schedule validation

#### Client folder structure (`client/src/`)

Within the React app:

- `client/src/pages/`
  - `loginPage.tsx` - calls `POST /auth` and navigates to `/main`
  - `schedulePage.tsx` - the main drag-and-drop scheduling experience (uses `useSchedule()`)
- `client/src/components/`

  - index page for access

### How to Think About “System Meaning”

In QICAS, the folder structure maps directly to responsibilities:

- **Routers** define _what URLs do_.
- **Controllers** define _how requests become service calls and HTTP responses_.
- **Services** define _the actual scheduling domain behavior_:
  - migration logic for years
  - rule storage/loading
  - validation logic and violation severity
- **DB schemas/types** define _the shape of the schedule and constraints_.
- **Frontend pages/hooks/components** define _how humans manipulate schedules_ and _how the UI interprets validation results_.

## Environment Variables

The backend reads environment variables from your `.env` file.

These variables currently exist in the codebase:

- `PORT`: Express server port (defaults to `3000` in `src/index.ts`)
- `MONGO_URI`: MongoDB connection string used by `src/db/connection.ts` and some seeders (defaults to `mongodb://localhost:27017/qicas`)
- `JWT_SECRET`: JWT signing/verifying secret used by `src/controllers/authController.ts` (defaults to `queensuniversity` if not set)

Cookie/session note: the JWT is stored in the browser cookie named `token` and scoped into requests as `faculty_id` by `verifyToken` middleware.

### Suggested Setup (high level)

Backend and frontend are split, so you typically run:

1. MongoDB (local or via `MONGO_URI`)
   - To run locally, follow [these steps](https://www.mongodb.com/docs/v7.0/tutorial/install-mongodb-on-ubuntu/#std-label-install-mdb-community-ubuntu) to install and run mongodb on your machine.
2. Backend:
   - `bun dev` (runs `src/index.ts` with hot reload)
3. Frontend:
   - `cd client && bun dev` or `npm run dev` (Vite)
